## 目的

Keystatic の編集画面はフォームの見た目だけを折りたたむ機能が限定的で、`fields.object` によるグループ化は frontmatter のデータ構造変更を伴う。
そのため当面は **フォームの入力順・説明文・運用ルール** で編集体験を整える。

---

## ブログ記事の入力順と役割

### 1. 基本情報

| フィールド | 用途 |
|---|---|
| タイトル | 記事タイトル。slug の元にもなるが、日本語タイトルの場合は slug を必ず英数で手入力する。 |
| URL slug | `/blog/<slug>/` の `<slug>`。英小文字・数字・ハイフン推奨。 |
| 概要 | 一覧・SEO・SNSで使う要約。1〜2文を目安。 |
| 公開日 | 一覧の並び順と公開可否判定に使う。未来日は予約投稿扱い（再ビルドが必要）。 |
| カテゴリ | 5つの大分類から選ぶ。細かい技術名や話題は tags に入れる。 |

### 2. 本文・画像

| フィールド | 用途 |
|---|---|
| 本文 | Markdoc 本文。本文中の画像は `public/images/blog` に保存される。 |
| ヒーロー画像 | カードサムネイル。16:9 推奨。 |
| OG画像 | SNS用のカスタム画像。未指定時は `/og/<slug>.png` が自動生成される。 |

### 3. 公開・詳細設定

| フィールド | 用途 |
|---|---|
| 下書き | チェック時は本番公開されない。 |
| 更新日 | 大きな追記・修正があった場合のみ設定。 |
| 著者 | 通常は空でよい。空の場合はサイト既定の著者名を使う。 |
| Canonical URL | 外部掲載など正規URLが別にある場合のみ設定。 |

## カテゴリ運用

カテゴリは「一覧での大分類・絞り込み」に使う。Keystatic では次の5つに固定する。

| value | 表示名 | 使い分け |
|---|---|---|
| `dev` | 開発・プログラミング | 言語、フレームワーク、ライブラリ、実装メモ |
| `hardware` | ハードウェア・電子工作 | Arduino、Raspberry Pi、半田、センサー等 |
| `tools` | ツール・インフラ | エディタ、クラウド、サーバ、運用、外部サービス |
| `reading` | 読書 | 技術書、電子書籍、読書メモ |
| `event` | イベント・参加記 | イベント参加、展示、コミュニティ関連 |

細かい話題（例: `Rust`, `Arduino`, `Mastodon`, `Maker Faire Tokyo`）は **tags** に入れる。

## タグ命名ルール

Keystatic の `tags` は現状フリーテキスト配列で、補完や重複チェックはない。表記揺れを防ぐため、次のルールを守る。

1. **既存タグを優先して再利用する**
   - 新規タグを作る前に、既存記事のタグや `/blog/tags/...` を確認する。
2. **固有名詞は公式表記に合わせる**
   - 例: `RaspberryPi`, `TypeScript`, `Maker Faire Tokyo`
3. **同義語を増やさない**
   - `Raspberry Pi` と `RaspberryPi`、`Unity` と `unity` のような揺れを避ける。
   - 既存記事で使われている表記を優先する。
4. **カテゴリ名をタグとして重複させない**
   - `開発・プログラミング` や `tools` のような大分類はタグにしない。
5. **1記事あたり 1〜5 個程度を目安にする**
   - 検索・関連記事判定に効く具体的なキーワードを選ぶ。

### 現在よく使われているタグ

- `Arduino`
- `RaspberryPi`
- `Python`
- `Rust`
- `Mastodon`
- `Maker Faire Tokyo`
- `読書`

## 将来検討: タグの relationship 化

タグ入力を本格的に補完 UI 化するには、タグを独立コレクション化して `fields.relationship` を使う必要がある。
ただしこの場合、タグは slug ベースの参照になり、既存の公開URL（例: `/blog/tags/Arduino/`）や表示名の扱いが変わる可能性がある。

そのため、現時点では URL 互換性を優先してフリーテキスト運用を継続し、必要になった段階で別 issue として設計する。
