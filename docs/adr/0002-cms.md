## コンテキスト

keroway.com は Astro 6 で構築された個人ポートフォリオ兼テクニカルブログです。M5 マイルストーン「CMS 環境」では、ブログ・Works コンテンツを CRUD できる仕組みを導入し、記事更新の摩擦を下げることを目指しています。

### 現状

- `src/content/blog/` に 52 記事（全て `.md`）、`src/content/works/` に 1 記事を保有
- `src/content.config.ts` に Zod スキーマ定義済み（`title / description / pubDate / heroImage / category / tags / draft / ogImage / author / canonicalUrl / readingTime` など）
- 画像は主に `/public/images/blog/{hash}.jpg` に保存。記事内では `![](https://img.youtube.com/vi/{ID}/0.jpg)](https://youtu.be/...)` パターンで YouTube サムネリンクも運用中
- ホスティングは **Vercel SSG**（`astro.config.mjs` に Cloudflare/Workers 設定はなし）
- CMS は未導入。記事追加は VS Code 等で `.md` を直接編集してコミット

### 要件

**必須要件**:

1. Markdown / MDX を WYSIWYG に近い UI で編集できること
2. 画像を UI からアップロードし、本文に挿入できること
3. 動画リンク（YouTube 等）からサムネイル表示ができること

**副次要件**:

- AI 補助執筆（Claude Code / ChatGPT 等）と相性が良いこと（Markdown ファイルが正本である方が望ましい）
- Release automation・PR ベースのレビューフローを既存ワークフローに組み込めること
- 学習コスト・新技術試用の魅力（Issue #29 の評価軸）
- ホスティングは当面 Vercel 継続を前提（Cloudflare 移行は Issue #15 で別途評価）

### ユーザー提示の 3 案

Issue #29 のディスカッションで、Reddit を起点とした以下 3 案がベースラインとして提示されました:

| 案 | 構成概要 |
|----|---------|
| **本命案** | Astro + Cloudflare Pages + Workers + D1 + R2 + 自作 SSR 管理画面 |
| **副案 1** | Astro + Cloudflare Pages（or 既存 Vercel）+ Git-based CMS（Decap / Pages CMS） |
| **副案 2** | Astro + Markdown ファイル直編集（CMS なし） |

本 ADR では、これら 3 案に加えて Issue #29 の正式候補リスト（Keystatic / TinaCMS / Sanity / Payload v3 / Astro Studio / Directus / Strapi）を含めて評価しました。

## 決定事項

**Keystatic（`@keystatic/astro`）を採用する。** 副案 1（Git-based CMS）の具体化として位置付けます。

採用構成:

- **配置**: 既存リポジトリ（単一リポ）のルートに `keystatic.config.ts` を配置
- **コレクション**: `src/content.config.ts` の Zod スキーマを Keystatic `collection` 定義に写像。既存 52 記事はパス・拡張子・frontmatter とも変更せずに継続利用
- **編集モード**:
  - ローカル: `/keystatic` で開発時に起動（`pnpm run dev` 中のみアクセス可能）
  - リモート: Keystatic Cloud または GitHub App 経由のブランチモード（PR ベース）。本番運用は Vercel Preview Deployment + GitHub PR レビューに乗せる
- **メディア**: 当面はリポジトリ内 `public/images/blog/` に保存（既存運用と同一）。外部ストレージ移行は別 ADR（`0004-media-storage.md`, Issue #97）に分離
- **動画サムネ**: 既存の YouTube サムネリンクパターンを継続。Keystatic の `text` フィールドで Markdown として記述する。専用フィールドの追加は PoC 後に判断

## 採用理由（Why Keystatic）

1. **Astro 公式統合**: `@keystatic/astro` インテグレーションがあり、Content Collections と直結する設計。既存 schema と運用を破壊しない
2. **既存資産の継続性**: 記事は `src/content/blog/*.md` のまま保持。Keystatic は Zod 互換に近いフィールド定義を持ち、`src/content.config.ts` の写像コストが低い
3. **必須要件の標準装備**: Markdown / MDX エディタ、画像アップロード（リポジトリへ自動コミット）、フィールドベースの frontmatter 編集をデフォルトで備える
4. **AI 補助執筆との相性**: 正本は Git 上の Markdown のままなので、Claude Code 等で記事を生成・編集してコミットするフローと並行できる（同一ファイルを CMS でも編集可）
5. **TypeScript-first**: `astro/tsconfigs/strict` 構成と整合。`keystatic.config.ts` を型安全に書ける
6. **段階導入**: ローカル限定 → Keystatic Cloud / GitHub App によるリモート編集 → 必要なら R2 等への画像外出し、と段階的に拡張できる
7. **コスト**: OSS でセルフホスト可、Cloud は任意。Vercel SSG と完全併用可で追加ホスティング不要

## 却下した候補

### 本命案（Cloudflare D1 + R2 + 自作 admin）

**却下理由**:

1. **必須要件に対して過剰**: 本命案が解く問題（独自管理画面・DB ベース CRUD・SSR ワークフロー）は、現規模（記事 52 件、書き手 1 名）では費用対効果が低い。Keystatic で必須要件は全て満たせる
2. **ホスティング移行コスト**: Vercel SSG → Cloudflare Pages + Workers の移行は Issue #15（SSR / Adapter 評価）の決定範囲。本 ADR の独立した決定として束ねるのはスコープ過大
3. **自作 admin の継続保守負担**: 管理 UI の自作は初期実装＋認証・権限・WYSIWYG エディタ・画像アップローダ等の継続保守が発生し、コンテンツ運用に集中する余力を削ぐ
4. **段階導入が可能**: Phase 3 として将来再検討する道は閉ざさない（後述「段階導入ロードマップ」参照）

### Decap CMS

**却下理由**:

1. **メンテナンス活性の低下**: 旧 Netlify CMS からのフォークだが、近年のリリース頻度・エコシステムの活性が Keystatic と比べて鈍化傾向
2. **Astro 専用設計ではない**: Hugo / Jekyll / Gatsby 等を含む汎用 Git CMS で、Astro Content Collections のスキーマと自然に同期する仕組みはない。設定を二重管理しがち
3. **認証フローが Netlify 前提**: Netlify Identity 連携が前提に近く、Vercel + GitHub OAuth で組む場合に Git Gateway を別途立てる必要がある

### Pages CMS

**却下理由**:

1. **エコシステム規模**: 軽量で動作はするが、Keystatic と比べてコミュニティ・実装事例・型支援が薄い
2. **Astro との型連携が手作業**: Content Collections のスキーマ写像は手書きになる

### TinaCMS

**却下理由**:

1. **Next.js / React 偏重**: Astro での利用は技術的に可能だが、ドキュメントの中心は Next.js。Astro 6 + UnoCSS 構成で詰まったときの参照情報が薄い
2. **複雑性**: Git + GraphQL レイヤと TinaCloud の有償プランが入り組んでおり、現在の規模に対して過剰

### Sanity

**却下理由**:

1. **SaaS / 外部 DB 依存**: 記事の正本が Sanity Cloud 側になり、AI 補助執筆や Git ベースの diff レビューと相性が悪い
2. **既存記事の移行コスト**: 52 記事を一括移行する必要があり、本 ADR の「最小限の変更で必須要件を満たす」方針から外れる

### Payload v3

**却下理由**:

1. **Next.js ネイティブ**: v3 系は Next.js 統合が前提。API として外から叩く構成は可能だが、Astro プロジェクト単体に置く CMS としては重い
2. **DB 必須**: Mongo / Postgres の運用が必要で、Vercel SSG のシンプルさを失う

### Astro Studio / Astro DB

**却下理由**:

1. **正本が DB 化**: Markdown ファイルの利点（AI 補助執筆・Git diff・grep）を失う
2. **サービス継続性**: Astro Studio のホスト型サービスは状況が変化しているため、本ブログのような長期運用前提のコンテンツの保管先として現時点では推奨しづらい

### Directus / Strapi

**却下理由**:

1. **ヘッドレス汎用 CMS**: 大規模・多言語・複数モデルを想定した設計で、ブログ 1 コレクションには過剰
2. **Self-host 負担**: DB + バックエンドプロセスの常時稼働が必要で、Vercel SSG の運用コスト 0 を毀損

### 副案 2（CMS なし）

**却下理由**:

1. **必須要件未達**: 「Markdown 編集 UI」「画像添付ワークフロー」が満たせない。VS Code 直編集に依存するため、出先・モバイルからの軽微な修正に弱い
2. ただし副案 2 の運用自体は **継続して併用可能**。Keystatic と直接ファイル編集は同じ正本（リポジトリ内 `.md`）を共有するため、両立できる点は本決定の安全弁となる

## 関連 Issue への影響

### #30 リポジトリ構成

- **方針**: monorepo 化は不要。単一リポジトリ継続で十分。
- **根拠**: Keystatic は `keystatic.config.ts` をルートに置くだけで動作し、`apps/web` / `apps/cms` / `packages/content-schema` のような分割を要求しない。
- **接続**: 詳細は別 ADR `0005-repo-structure.md`（Issue #88）で確定。本 ADR からの推奨は「単一リポ継続」。

### #32 スキーマ拡張

- **影響**: 既存 Zod スキーマを Keystatic コレクション定義に写像する作業が発生。
- **同期戦略**: 当面は手動で 2 か所同期。将来は `keystatic.config.ts` ↔ `content.config.ts` の単一情報源化（コード生成 or 共有モジュール）を検討。
- **PoC で確認すべき点**（Issue #86）: `category`（string）、`tags`（array of string）、`heroImage`（string パス）、`draft`（boolean）、`pubDate`（date）の各フィールドが Keystatic UI 上で違和感なく編集できるか。

### #34 プレビュー / デプロイフロー

- **方針**: Keystatic の **branch mode** を採用し、CMS 上での編集を GitHub ブランチへの push に変換、Vercel Preview Deployment で確認、PR マージで本番反映する。
- **下書き**: 既存スキーマの `draft: boolean` を活用。`getCollection('blog', ({ data }) => !data.draft || import.meta.env.DEV)` のフィルタを SSG 時に適用する案。
- **詳細**: Issue #100（Vercel Preview と CMS の連携検証）で実機検証。フロー図は Issue #99（`docs/cms-flow.md`）で作成。

### #15 レンダリング戦略

- **整合性**: 本決定は **Vercel SSG の継続を前提**としており、レンダリング戦略の選択肢（SSG / SSR / ISR / Hybrid）に対して中立。
- **理由**: Keystatic 自体は静的サイト出力に影響を与えない。`/keystatic` 管理画面はローカル開発時のみ動作し、本番ビルドからは除外可能。
- **将来 SSR 化する場合**: Keystatic と SSR は並存可能（Keystatic GitHub App でリモート編集する構成は Vercel/Cloudflare いずれでも成立）。

### #33 メディアストレージ

- **当面の方針**: 既存と同じく `public/images/blog/` にコミット。Keystatic の `image` フィールドでアップロード先を指定する。
- **将来**: Cloudflare R2 / Vercel Blob / Cloudinary 等への外部化は別 ADR `0004-media-storage.md`（Issue #97）で扱う。Keystatic は外部ストレージへの差し替えに対応している（custom image storage）。

### 動画リンクサムネ要件

- **現状運用**: `![](https://img.youtube.com/vi/{ID}/0.jpg)](https://youtu.be/{ID})` の Markdown パターン
- **Keystatic 上の表現**: `document`（Markdown）フィールドでそのまま記述可能
- **将来検討**: `videoUrl` カスタムフィールドを追加し、ビルド時に `astro-embed` / rehype プラグインで oEmbed 経由でサムネを自動取得する案。PoC 後に必要性を判断（必須要件は既存パターンで充足するため、優先度は中）

## 段階導入ロードマップ

ユーザー提示の Phase 1/2/3 を取り込み、以下のロードマップで進めます。

| Phase | 構成 | 状態 |
|-------|------|------|
| **Phase 1** | Astro + Vercel + Markdown 直編集 | 現状 |
| **Phase 2** | Phase 1 + **Keystatic**（本 ADR の対象） | 本 ADR で採用、Issue #86 で PoC |
| **Phase 3** | Phase 2 + 必要に応じて R2 / SSR / 自作 admin 拡張 | 将来再検討（Issue #15 / #97 と連動） |

Phase 3 への移行は、以下のトリガーが揃った場合に再評価する想定:

- 非技術者の編集者が継続的に増える
- 画像/メディア量がリポジトリ管理の限界（数 GB 規模）を超える
- 独自ワークフロー（ドラフト承認・公開予約の高度化・AI 記事生成 UI）が必要になる

---

## 結果

Keystatic を採用することで以下が実現できます:

- 必須要件（Markdown 編集 UI / 画像添付 / 動画サムネ）を最小実装コストで充足
- 既存 52 記事・スキーマ・Vercel SSG を変更せずに導入できる
- AI 補助執筆と CMS 編集が同じ Markdown 正本を共有でき、両立する
- ホスティング移行（Issue #15）と独立した決定として、CMS だけ先行導入できる
- Phase 3 への発展余地（R2 / SSR / 自作 admin）を閉ざさない
