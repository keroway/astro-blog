# 0019: 記事フィードバックチャネルは当面導入しない

- Status: Accepted
- Date: 2026-07-03
- Issue: #473

## コンテキスト

現状の記事詳細には共有導線 (`PostShare`) はあるが、コメント・リアクション・Webmention のような受信型フィードバックチャネルは無い。

導入候補は以下だった。

1. Webmention — IndieWeb 標準で静的サイト思想と相性が良いが、webmention.io 等の外部依存とビルド時取得処理が増える。
2. giscus — GitHub Discussions 連携でエンジニア読者層と相性が良いが、サードパーティ iframe と CSP 調整が必要。
3. 見送り — 静的サイトのシンプルさ、CSP、プライバシー面の負荷を維持する。

既に CSP は enforcing で運用しており、外部 iframe / script / API 依存を追加する場合はセキュリティヘッダー・プライバシー説明・運用監視を同時に増やす必要がある。

## 決定事項

**当面、記事へのコメント / Webmention / リアクション機能は導入しない。**

Rationale:

- 現時点では読者からのコメント需要を示す明確なシグナルが無い。
- giscus は CSP とサードパーティ iframe の調整が必要で、サイトの軽量性・静的性に対してコストが大きい。
- Webmention は思想面では合うが、外部サービス依存・取得キャッシュ・スパム対策・表示 UI が必要になる。
- まずは PostShare、RSS/feed.xml、検索・導線改善を優先し、反応が必要な場合は GitHub / SNS など既存チャネルへ誘導する。

## 却下した候補

### Webmention

- webmention.io 等への依存とビルド時取得処理が必要。
- スパムや削除済み mention の扱いを決める必要がある。
- 受信件数が少ない段階では UI / 運用コストに見合いにくい。

### giscus

- iframe / script 追加により CSP 調整が必要。
- GitHub アカウント前提のコメント体験になり、個人サイトとしての軽さを損ないやすい。
- コメントモデレーションが GitHub Discussions 運用に依存する。

## Revisit When

- 記事への反応・質問が月 5 件以上、既存チャネル (SNS / GitHub / メール等) で継続して発生する。
- CSP / プライバシーポリシーを更新する別要件が発生し、外部 iframe / script の許可方針を再検討する。
- Webmention を静的に安全に取り込む運用 (キャッシュ、モデレーション、スパム除去) を小さく実装できる見通しが立つ。
