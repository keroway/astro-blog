## Why this matters

The first blog listing page has a normal self-canonical URL, but page 2 and later currently receive two canonical tags: one generated by `BaseHead` for the current paginated URL, and one inserted by the blog page pointing to `/blog/`. Duplicate conflicting canonicals create avoidable SEO ambiguity. The fix is small: pass the intended canonical URL through the existing layout/head prop path instead of adding a second `<link rel="canonical">` via a slot.

## Steps

### Step 1: Route the non-first canonical through `SiteLayout`

In `src/pages/blog/[...page].astro`, create a canonical variable near `isFirstPage`:

```ts
const canonicalUrl = isFirstPage
  ? undefined
  : new URL('/blog/', Astro.site).href;
```

Then pass it to `SiteLayout`:

```astro
<SiteLayout
  title={title}
  description={SITE_DESCRIPTION}
  mainClass="blog-index"
  canonicalUrl={canonicalUrl}
>
```

Remove only the duplicate `<link rel="canonical" ...>` from the `Fragment slot="head"`. Keep the `robots` meta for non-first pages.

Target non-first block:

```astro
{!isFirstPage && (
  <Fragment slot="head">
    <meta name="robots" content="noindex, follow" />
  </Fragment>
)}
```

### Step 2: Add regression coverage for page 2 canonical uniqueness

Add a test to `tests/playwright/basic.spec.ts` under `test.describe("SEO: canonical URL", ...)`.

Test requirements:

- Navigate to `/blog/2/`.
- Assert there is exactly one `link[rel="canonical"]`.
- Assert its `href` is `https://keroway.com/blog/`.
- Assert `meta[name="robots"]` has content `noindex, follow`.

Example target shape:

```ts
test("paginated blog listing has one canonical to the first page", async ({ page }) => {
  await page.goto("/blog/2/");
  const canonical = page.locator('link[rel="canonical"]');
  await expect(canonical).toHaveCount(1);
  await expect(canonical).toHaveAttribute("href", "https://keroway.com/blog/");
  await expect(page.locator('meta[name="robots"]')).toHaveAttribute(
    "content",
    "noindex, follow"
  );
});
```

If the local dev server normalizes the URL without trailing slash but the generated canonical is still exactly `https://keroway.com/blog/`, keep the exact assertion above; it tests the intended SEO output.

## Maintenance notes

If future paginated pages need custom canonical behavior, prefer passing `canonicalUrl` into `SiteLayout` instead of injecting another canonical link through the `head` slot. Reviewers should check the rendered `<head>` rather than only the Astro source: this bug was caused by two individually reasonable code paths composing into duplicate output.
